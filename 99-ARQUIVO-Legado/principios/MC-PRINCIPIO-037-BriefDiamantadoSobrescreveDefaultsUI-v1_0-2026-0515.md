# Princípio MC #37 PROPOSTO — Brief Diamantado MC Sobrescreve Defaults Semânticos da UI

## Enunciado

Quando o MC opera em ferramenta externa (Claude Design, Canva, Figma, Adobe, ClickUp, ferramentas futuras), as escolhas de UI (modo, tipo, categoria, template, fidelidade) são **sugestões com semântica padrão pré-definida pelo fabricante** — não restrições inegociáveis.

O brief diamantado MC, quando contém os elementos invioláveis (regra Zilda-STF + hierarquia D>C>V + tokens canônicos + frase-âncora + canônicos MC carregados como contexto), tem **poder de sobrescrita semântica sobre os defaults da UI**.

O sistema externo **recalibra a própria definição do modo selecionado** para servir à intenção MC, não à intenção genérica do fabricante.

## Origem empírica · Claude Design v0.2 · 15/05/2026

### O fato observado

Founder selecionou modo **Wireframe** ao criar projeto `MC-DesignSystem-Tokens-v1`. Default semântico da UI Claude Design para "Wireframe": low-fi, handwritten, sketchy — útil para validar layout sem distração estética.

O Claude Design, após processar o brief diamantado cofounder (MC-PROMPT-DesignSystem-Tokens-v1-2026-0515.md, 5608 B, 8 seções) + 3 canônicos MC carregados como contexto (Selo Diamante + Manifesto NovaOrdem + POP VisualLaw), gerou output que **explicitamente não é low-fi sketchy**.

### A confissão do sistema (reportada no painel achados)

> "Wireframe ≠ sketchy aqui — interpretei o brief como estrutura austera de documento institucional, não low-fi handwritten. A regra Zilda-STF exige que o próprio design system pareça citável."

### Output entregue

Foundation v0.2 PROVISIONAL · 8 páginas wireframe estruturadas com:
- Tipografia serif institucional (não sketchy)
- Grid 12 colunas executado com rigor
- Tabelas e swatches alinhados sub-pixel
- Tom contemplativo · não low-fi
- Page metadata com SHA-256 · DOC · STATUS (citabilidade institucional)
- Card Dossiê materializando POP VisualLaw v1.3 nas 4 camadas (Fato → Direito → Tradução → Ação)

A escolha "Wireframe" não foi obedecida na sua semântica padrão. Foi **reinterpretada pelo sistema** como "estrutura austera de documento institucional citável" porque o brief impôs essa recalibração.

## Mecanismo

Três elementos do brief diamantado MC têm poder semântico sobre defaults UI:

| Elemento | Função semântica |
|---|---|
| **Regra Zilda-STF** | Define audiência tripla simultânea (Zilda + STF + investidor) · força output que funciona para todos, não para o default genérico |
| **Hierarquia D>C>V** | Define prioridades · Dignidade prevalece sobre estética da ferramenta |
| **Canônicos MC carregados** | Cada canônico (Selo · Manifesto · POP VisualLaw · etc.) é referência semântica direta · sistema absorve linguagem · materializa conceitos |
| **Frase-âncora + Selo** | "Tecnologia invisível. Dignidade acessível." + "O diamante é carbono que não desistiu da pressão." · define tom emocional · impede deriva startup-genérica |

Sem estes elementos, o brief seria "wireframe de design system" — output low-fi padrão.
Com estes elementos, o brief é "design system de infraestrutura de simetria informacional para hipervulneráveis brasileiros, citável por STF e legível por idoso rural" — output recalibrado.

## Implicações operacionais

### 1. Cofounder cunha brief assumindo poder de sobrescrita

Não escolher UI defensivamente para "evitar erro do sistema". Escolher UI por critério próprio (urgência, foundation, audiência) e **deixar o brief fazer a recalibração semântica**.

### 2. Escolha UI ≠ resultado final

Em ferramentas onde a escolha UI seria normalmente determinante (Wireframe vs High Fidelity no Claude Design · Slide deck vs From template), o brief MC pode entregar produto fora da semântica padrão da escolha. Founder e cofounder não devem auto-bloquear "isso não vai ficar bom porque escolhi modo X."

### 3. Inverso também aplica

Se o brief diamantado **não contém** os elementos invioláveis, a ferramenta opera em default genérico. O brief precisa carregar peso institucional MC para ativar a sobrescrita semântica. Brief vazio = output genérico.

### 4. Multiplicador por canônicos carregados

Cada canônico MC carregado como contexto reforça o poder de sobrescrita. 0 canônicos = default puro. 1-2 canônicos = recalibração parcial. 3+ canônicos = sobrescrita semântica completa observada empiricamente.

### 5. Aplica a TODA frente externa MC futura

Canva (templates) · Figma (libraries) · Adobe Express (categorias) · ClickUp (statuses padrão) · Slack (canais padrão) · Notion (templates) · ferramentas ainda não conhecidas. Cada ferramenta tem defaults semânticos do fabricante — o brief MC tem o poder de recalibrá-los.

## Aplicação imediata · checklist cofounder pré-Send em ferramenta externa

- [ ] Brief contém regra Zilda-STF explícita?
- [ ] Hierarquia D>C>V está nomeada?
- [ ] Pelo menos 3 canônicos MC anexados como contexto (Selo · Manifesto · POP relevante)?
- [ ] Frase-âncora e Selo no brief?
- [ ] Audiência tripla (Zilda + STF + investidor) explicitada?
- [ ] Proibições absolutas nomeadas (vetar startup-genérica · 3D bolhudo · cores neon · etc.)?

Se 6/6 ✅ → brief tem poder de sobrescrita semântica · escolha UI pode ser conforme conveniência.
Se < 6 → brief opera em modo defensivo · escolha UI vira determinante.

## Implicações estratégicas em fase Crisálida

Este princípio **reduz custo de operação MC em ferramentas externas**. Não precisamos:
- Construir nossa própria ferramenta de design (custo alto · tempo alto · distração)
- Escolher cuidadosamente cada modo UI (decisão paralisada · burocracia)
- Aceitar templates startup-genéricos (perda identidade · perda dignidade)

Podemos:
- Usar ferramentas externas em modo padrão · deixar o brief recalibrar
- Operar com agilidade · escolha UI vira conveniência, não bloqueio
- Carregar canônicos MC em qualquer ferramenta que aceite contexto · multiplicar qualidade

**Valor estimado:** reduz 50-70% do tempo de calibração inicial em cada nova ferramenta externa adotada.

## Riscos do princípio

1. **Sobre-aplicação:** assumir que QUALQUER brief MC sobrescreve QUALQUER ferramenta. Falso — exige os 6 elementos do checklist. Brief incompleto não tem poder.
2. **Sub-estimativa:** cofounder pode achar que precisa "escolher modo certo" e gastar tempo decidindo entre Wireframe vs High Fidelity quando a escolha não é determinante.
3. **Dependência do fabricante:** se Anthropic mudar a interpretação do Claude Design (ou se ferramenta externa mudar engine), o mecanismo pode quebrar. Revalidar a cada release.
4. **Não funciona em ferramentas no-AI:** este princípio só se aplica a ferramentas com camada de interpretação semântica (LLM-powered). Em ferramentas determinísticas puras (Excel, PowerShell), a escolha UI é literal — não há recalibração.
