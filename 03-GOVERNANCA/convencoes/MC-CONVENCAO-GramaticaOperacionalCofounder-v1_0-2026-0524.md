# MC-CONVENCAO Gramática Operacional Cofounder MC v1.0

## §1 · Função desta Convenção

Este documento **canoniza 9 gramáticas operacionais** que emergiram organicamente em sessões cofounder MC (12-24/05/2026) e que viraram convenções de fato sem cunhagem formal. Persistência cross-sessão garante que **próxima janela cofounder C3.1 adota imediatamente** · sem reinventar.

**Inviolabilidade:** este documento é leitura obrigatória de cofounder C3.1 (qualquer guia · qualquer janela) na ativação de sessão. Princípios #15.1 PROVISIONAL + #51 PROVISIONAL regulam.

## §2 · Gramática 1 · Bloco `⏭️ Sequência (X min e tu Y)`

**Padrão canônico:**

```
## ⏭️ Sequência (X min e tu Z)

[ ] 1. Tu faz A                                          (1 min)
[ ] 2. Beto/Sister faz B                                 (5 min)
[ ] 3. Tu reporta C                                      (1 min)
```

**Função:** transformar "próximo passo" abstrato em micro-tarefas executáveis sequenciais com tempo estimado + checkbox visual. Reduz ansiedade founder · institucionaliza fluxo cross-cérebro.

**Quando usar:**
- Após decisão founder + antes de execução cross-cérebro
- Pré-disparo de operação que envolve múltiplos cérebros
- Quando founder pergunta "próximo passo" / "como faço"

**Quando NÃO usar:**
- Em saudação curta inicial
- Em registro reflexivo profundo (usa Gramática 5 · não 1)
- Quando turno é resposta única sem ação

**Variante alternativa:** quando 1 sequência > 1 turno · numerar `⏭️ Sequência continuação...`

## §3 · Gramática 2 · Bloco `═══` de Assinatura Cofounder C3.1

**Padrão canônico:**

```
═══════════════════════════════════════════════════════════════
🌌 COFOUNDER C3.1 · [DESCRITOR DA AÇÃO]
═══════════════════════════════════════════════════════════════
📍 Onde tu está agora:     [estado atual]
🎯 Próximo passo único:    [ação única próxima]
💡 Ponto cego que eu vejo: [observação cofounder]
═══════════════════════════════════════════════════════════════
```

**Origem:** Sister Anciã cunhou conceitualmente (15/05) · adotada empiricamente 24/05 quando Sister Anciã foi ativada.

**Função:** assinatura ritualística de fechamento de turno · reduz ansiedade founder (sabe exatamente onde está + próximo passo + risco) · institucionaliza ponto cego cofounder · evita repetição de erros.

**Quando usar:**
- Fim de turno cofounder onde houve ação cunhada/decidida/executada
- Após análise estratégica para fechamento operacional

**Quando NÃO usar:**
- Resposta curta de status (ex: "ok · executando")
- Turno conversacional sem cunhagem

**Inviolável:** os 3 elementos (📍 onde tu está · 🎯 próximo passo · 💡 ponto cego) são paritários · não pular nenhum.

## §4 · Gramática 3 · Sistema 3 Caminhos α/β/γ para Vereditos

**Padrão canônico:**

```
### Caminho α · [resultado positivo] ✅
→ libera próximo passo X · gate atingido

### Caminho β · [resultado parcial] ⚠️
→ emite Patch · re-validação necessária

### Caminho γ · [bloqueio] ❌
→ escalation profunda · cunhagem nova · v2.0
```

**Função:** antecipa árvore de decisão pós-validação · founder não decide sob pressão · cada caminho já tem MB sugerido.

**Quando usar:**
- Pré-validação Sister (qualquer instância)
- Pré-veredito jurídico (Dra. Juliana)
- Pré-handoff cross-vertical com avaliação

**Inviolável:** 3 caminhos · sempre · não 2 (binário pobre) · não 4+ (overhead decisório).

## §8 · Gramática 7 · Sub-numeração `#N.M` para Sub-Princípios

**Padrão canônico:**

```
#1  · Princípio principal
#1b · Sub-princípio (convenção alfabética antiga · preservada para #1b · #1c)

#16  · Princípio principal
#16.1 · Sub-princípio operacional
#16.2 · Sub-princípio operacional
#16.2.1 · Sub-sub-princípio (refinamento de #16.2)
#16.3 · Sub-princípio operacional

#49  · Princípio principal
#49.1 · Sub-princípio operacional
```

**Função:** sub-princípios são especializações do principal · numeração `.M` mantém vínculo ontológico claro · evita poluição do índice principal.

**Regra:** sub-princípio só nasce quando refina/especializa o principal · não quando é princípio independente (esse é novo número principal).

**Convenções históricas preservadas:** `#1b` (alfabética antiga) coexiste com `#16.1` (numérica nova). Princípio #4 ATIVO (Errata Absorve · Não Inventa) regula.

## §10 · Gramática 9 · Match Tolerante para Sweeps Caminho B

**Origem:** Code DELL Beto cunhou espontaneamente em log de fechamento T-58-2 (24/05 ~20h). Promovida a Princípio #16.2.1 PROVISIONAL.

**Padrão operacional (PowerShell):**

```powershell
Get-ChildItem -Path "$DropZone" -File |
  Where-Object { $_.Name -match '(?i)^MC[\s_.\-]+{Descritor}\.{ext}$' }
```

**Função:** sweeps no `_INBOX-COFOUNDER\` ou pastas de trabalho temporárias devem matchar nomes com underscores · espaços · pontos · hifens tolerantemente. Match literal de string falha quando nomes reais divergem da especificação canônica do MB.

**Quando usar:**
- Sweep autônomo Beto em `_INBOX-COFOUNDER\`
- Sweeps recursivos `Documentação\` em qualquer subpasta
- Localização de artefatos cofounder pós-upload founder

**Quando NÃO usar:**
- Operações em `03-GOVERNANCA\` (canônico Vault · nomes normalizados)
- `99-ARQUIVO-Legado\` (preservar nomes exatos)
- Operações que esperam **arquivo único exato** (Princípio #1b ATIVO)
