## 2. CAMPO DE ADVERSIDADE — RISCOS, GAPS, CONSTRAINTS E ARMADILHAS

Mapeado a partir de artefatos (relatórios, playbooks, colheitas, ADRs, encerramentos de fase, selos, prompts de choque de realidade).

### 2.1 Constraints Técnicos / Econômicos (Alta Prioridade)
- **Custo de Tokens (Claude Opus)**: Input $5/M barato; Output $25/M caro. Força arquitetura de "Tweaks" (sliders locais JS que não re-geram DOM completo), /EDITMODE-BEGIN/ JSON para persistir estado paramétrico sem re-inferência, comentários inline para micro-ajustes. Qualquer build que force reescrita massiva de código é anti-econômico e latente.
- **Janela/Orçamento Finito NotebookLM**: 300 fontes/caderno, 150M palavras — "orçamento" que não pode ser desperdiçado com redundância ou fontes fracas. 500 queries chat/dia, 20 Deep Research. Se encher com lixo, "apenas amplifica o ruído".
- **Processamento Visual vs Texto**: 4.7/4.8 forte em visão (2576px) para Self-QA screenshot loop, mas render no sandbox consome. Handoff para code é o destruidor de atrito tradicional (designer→dev).
- **Atualizações Rápidas de Modelo**: 4.7 (16/04, motor visual) vs 4.8 (28/05, honestidade 4x, fast mode). Selos precisam ressalvas e reabertura condicional. Risco de drift de versão no vault/CLAUDE.md.
- **Limites de Cota e Privacidade**: NotebookLM Pro bom (não treina em dados Workspace), mas quotas diárias forçam priorização cirúrgica. Claude Design/Cowork em preview/research têm barras de progresso de créditos.

### 2.2 Qualidade e Curadoria (Core do Método)
- **Diluição por Acúmulo**: Gargalo não é garimpar — é decidir o que NÃO entra. Fontes "mais ou menos" contaminam downstream. Compasso (3 cortes) é a defesa; sem ele volta a "listas superficiais e colheitas pobres".
- **Falso Positivo de "Ensina-Fazendo"**: Vídeos que "falam mas não mostram código rodando" ou pulam passos críticos (ex: fermentação na panificação). Balote obrigatório.
- **Evidência Circunstancial vs Literal**: Muitos selos (ex: motor 4.7 do Design) são circunstanciais (data de lançamento + specs). Linha de Evidência exige registrar "o que NÃO prova" + condição de reabertura. Risco de overclaim.
- **Redundância de Ângulo**: Múltiplos tutoriais hello-world ou SEO genérico sobre o mesmo. Filtro de unicidade implícito no Compasso.

### 2.3 Filtro Brasil / Hipervulneráveis / Conformidade (Missão-Específico)
- **Recorte Nacional Obrigatório**: Stacks US-centric (Claude for Legal, etc.) não convergem com realidade causal brasileira (INSS, gov.br, CADÚNICO, processos previdenciários, hipervulneráveis com baixa literacia digital?). Colheitas devem priorizar replicabilidade local.
- **LGPD + Dados Sensíveis + ANPD/SCCs**: CPF, CNIS, CID são sensíveis. MC-ACI tem LGPD linha vermelha por camada (Princípio #21). ADR-012 em provisional aguardando respostas Sales (Google/Anthropic/AWS para Brazilian SCCs / residência sa-east-1) + parecer Juliana 10/10/2026. Risco alto de transferência internacional de dado Art.11 CID.
- **Prova para Hipervulneráveis**: Tudo deve servir ao "campo de missão" (D>C>V). Decisões técnicas impactam 220M potenciais, com foco nos mais vulneráveis. "Não é só ferramenta — é infraestrutura de acesso digital".
- **"Analfabeto Digital" Founder como Feature, não Bug**: O sistema existe para amplificar o julgamento do leme sem substituí-lo. Qualquer instância que devolva "ideias soltas" ou exija trabalho manual pesado do Founder falha o contrato.

### 2.4 Integração e Operação Local (Sync e Ferramentas)
- **Cloud vs Local Fricção**: NotebookLM (Google cloud) + Claude web vs Obsidian (Drive sync) + Claude Code (terminal/MCP local, possivelmente Mac Mini M4 para C4). Precisa de:
  - Limpeza automática de transcrições YT (metadados, timestamps, ruído) antes de injeção.
  - Filtros RegEx / frontmatter YAML para isolamento de contexto por projeto (evitar poluição cross-vault).
  - Triggers PWD, servidores MCP (ex: 22360), REST local para ingestão autônoma.
  - Evitar depósito de binários/PDFs pesados no vault (destrói qualidade do contexto para Claude Code).
- **Persistência de Estado "Invisível"**: Tweaks e /EDITMODE são geniais para contornar nuvem, mas precisam ser extraídos/replicados em builds locais para que Claude Code produza código "empírico" sem AI slop.
- **Histórico de Misalignment em Sessões**: "Outra guia" (Claude?) entregava listas estáticas de SEO em vez de respeitar protocolo. Exigiu "Prompt 01: Choque de Realidade na Arquitetura" + "Prompt 02: Comando da Colheita Alfa". Risco recorrente se instâncias não forem calibradas com o contrato canônico.
- **Versionamento e Auditabilidade**: Git messages devem contar a história. Selos/playbooks precisam linkar de volta às fontes + NotebookLM + evidência. Sem isso, "volta a ser terminal com LLM".
- **Escalação de Esquadrão**: Caçador/Falcão/Arquiteto/Sniper precisam de prompts de calibração cirúrgica por função. Sem "contrato entre IAs", o sistema degrada.

### 2.5 Riscos de Transição de Fase (do Embate)
- Cada "encerramento de fase" (Google→Perplexity, Perplexity→Grok) revela que o motor anterior não entregava "formato de saída certo" + "pergunta boa" alinhado ao Compasso. Risco de repetir se Grok não for 100% disciplinado (nada de listas genéricas; sempre REG/ESC/JOG + bloco pronto + .md estruturado).
- Dependência de Founder para colar blocos manualmente entre ferramentas (risco de erro humano; oportunidade para automação build).
- C4 Cowork "em nascimento" — quando nascer, o MC-RAG precisa alimentar ele também sem fricção.

### 2.6 Matriz de Adversidade (Priorizada)
| Adversidade | Impacto | Probabilidade | Mitigação Atual / Proposta Grok Build |
|-------------|---------|---------------|---------------------------------------|
| Token waste em re-gens | Alto (custo + latência) | Alta | Tweaks + local param (já mapeado); Grok Build prototipa scripts que extraem /EDITMODE e geram builds "empíricos". |

**Risco Supremo:** Qualquer instância (incluindo futura Grok Build) que "substitua o julgamento do Founder" ou gaste tempo do leme em mecânica. O ecossistema amplifica o princípio — não o substitui.
