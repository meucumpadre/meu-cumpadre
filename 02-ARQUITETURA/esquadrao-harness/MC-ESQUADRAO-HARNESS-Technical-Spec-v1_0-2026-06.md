## 2. Architecture Overview (Inspired by Anthropic + MC ACI + Study)

**High-Level Model (following Anthropic Managed Agents + Harness principles + MC ACI instances):**

The Harness is the "product" layer that sits between the models (Claude via Code/Design/Managed Agents) and the tools/context.

**Core Components:**

1. **Lead Agent (Product Orchestrator)** — C3.1-like strategic brain for a product task. Breaks down "develop X feature of the harness" into sub-tasks. Delegates to specialists. Maintains overall plan and verification.

2. **Specialist Sub-Agents Fleet** (Zona Verde only):
   - **Design Specialist** (Claude Design heavy): Responsible for all product UX/UI layers of the harness and any MC internal tools it enables. Outputs ready for handoff to Code.
   - **Code Specialist** (Claude Code heavy): Implements the actual Python/TS/MCP/skills code for the harness components. Uses worktree isolation.
   - **Research & Market Intel Specialist**: Curates non-sensitive intelligence on Anthropic ecosystem, harness patterns, agent best practices (using MC-RAG firewalled cadernos + Compasso). Keeps the squad's context fresh.
   - **Verification & Antítese Specialist**: Runs the "landing validator" (adapted from our VTVL work), self-QA loops, adversarial reviews against the study notes (silence risk, Proof-First, no drift to vetado stacks, adherence to manifestos). Produces "pouso report".
   - **Harness Ops & Proactive Specialist**: Manages context compaction, session state, proactive workflows (Allie-style: agents that run "while we sleep" to update market radar, suggest harness improvements, pre-compile context packs).

3. **Shared Harness Infrastructure (the actual product being built):**
   - **Context & Memory Layer**: Firewalled MC-RAG (cadernos of Anthropic engineering, harness patterns, MC internal manifestos). CLAUDE.md per "project" (the harness itself is a project). Packs generated by our emitter tools.
   - **Extensibility Layer**: MCP (primary, per Anthropic direction), skills compostas, hooks, plugins. Designed so the harness can evolve without breaking as models improve (key lesson from Anthropic "harnesses encode assumptions that go stale").
   - **Orchestration & Delegation**: Sub-agent spawning with isolation, shared "file system" / workspace for artifacts (in vault or temp worktree).
   - **Verification & Landing System**: The adapted mc_vtvl_landing_validator logic — every major "pouso" (spec complete, code shipped, integration tested) gets a score + lessons + automatic suggestion for harness improvement.
   - **Proactive Engine**: Scheduler for overnight/background agents (market intel refresh, harness self-improvement proposals).

4. **Governance & Integration with ACI**:
   - The entire squad/harness is a C4-accelerated capability.
   - It enhances (does not replace) the orquestrador-mestre when used for internal product work.
   - All outputs go through internal equivalents of the 6 filters + Proof-First + "no vetado stacks" (ZapSign over DocuSign, etc.).
   - Human (Founder/C3.1) remains the sovereign leme for strategic direction and final gates.

**Data Flow (a typical "flight" / product development cycle):**
Founder/C3.1 gives high-level task ("Build the proactive workflow engine for the Squad Cockpit") 
→ Lead Agent plans + decomposes 
→ Delegates to Design + Research in parallel 
→ Code implements 
→ Verification runs landing check + produces report + lessons 
→ Lessons auto-fed back into Context Layer + suggested harness improvements 
→ "Booster" (improved harness components, updated packs, new verified patterns) is "refurbished" for the next flight.

This is exactly VTVL applied to internal product development.

## 3. Detailed Component Specifications (Ready for Claude Design + Claude Code)

### 3.1 The "Squad Cockpit" (Product Layer — Claude Design Primary)

This is the user-facing product of the harness (the interface through which humans direct and monitor the squad).

**Key Screens / Views to Design (detailed prompts in separate deliverable):**
- Mission Control (current flights/tasks, status, landing scores).
- Agent Fleet (Lead + Specialists status, workload, proactive jobs).
- Context Explorer (what the squad "knows" right now from firewalled RAG + packs; ability to inject new non-sensitive intel).
- Landing Reports (history of pousos with lessons learned and auto-applied harness improvements).
- Proactive Workflows (Allie-style scheduled agents; what is running overnight, results).
- Harness Self-Improvement Queue (suggestions from Verification + Research specialists).

**Non-Functional (from Anthropic harness research + study):**
- Excellent context management / compaction visible to user.
- Clear separation of "brain" (model decisions) from "hands" (tool actions via MCP).
- Auditability of every delegation and tool call (for internal "Proof-First").
- Graceful degradation and human override at any point (equivalent to Circuit Breaker).

### 3.2 The Orchestration Engine (Code Layer — Claude Code Primary)

Core loop (inspired by Anthropic "while loop" for agents + our existing orquestracao skill + VTVL landing):

```python
# Conceptual (full implementation in prototype)
while not mission_complete:
    context = context_manager.get_current_context(task, previous_landings)
    plan_or_action = lead_agent.think(context)  # or delegate
    if action:
        result = harness.execute_via_mcp_or_skill(action)  # with hooks
        verification = verification_engine.landing_check(result, task)
        context_manager.record_landing(verification)  # booster recovery
        if verification.requires_human:
            pause_for_human()
```

**Key Modules (to be implemented as separate files/skills):**
- orchestrator.py (Lead + delegation logic)
- context_manager.py (RAG firewalled + pack injection + compaction)
- verification_engine.py (landing validator adapted)
- mcp_client.py + skills_loader.py (extensibility)
- proactive_scheduler.py (background agents)
- worktree_manager.py (isolation for parallel specialists)

**Extensibility Contract (MCP + Skills + Hooks):**
Follow Anthropic's graduated context cost model. MCP for external/internal tools. Skills for domain instructions. Hooks for intercepting tool calls (e.g., auto-verify before execution).

- **Proof-First on capabilities**: Every claim about what the squad/harness "can do" in this spec or future docs must have source (official Anthropic or "inspired by X with honesty on limits").
